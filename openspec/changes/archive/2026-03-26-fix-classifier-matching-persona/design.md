## Context

系統實測時發現三個互相關聯的問題：

1. **牛肉麵誤判**：CLIP 的 positive/negative softmax 無法區分牛肉麵與魯肉飯，因為兩者共享深色滷汁、大塊肉的視覺特徵。負向提示詞列舉再多具體菜名也無法窮盡。
2. **Store matching 平手太難觸發**：正規化票數嚴格相等的機率極低，導致明顯相似的兩家仍強制選出一個，回應不誠實。
3. **平手時誤報配料**：平手情況下 `matched_store` 為 None，系統完全信任 CLIP 偵測的配料，但 CLIP 有視覺混淆（碎豬肉誤偵測為肉鬆）。

## Goals / Non-Goals

**Goals:**

- 有麵條的食物必須在分類前被排除，不進入 positive/negative 比較
- 平手邏輯應反映「視覺上確實難分難捨」的語意
- 平手時大叔描述的配料必須有店家背景知識支撐

**Non-Goals:**

- 不處理其他類別的誤判（如蛋炒飯、雞腿飯等）
- 不改變 CLIP 配料偵測的底層模型或向量計算方式
- 不引入其他 API 呼叫來提升分類準確率

## Decisions

### 麵條偵測 Gate

在進行 positive/negative softmax 之前，先對「是否有麵條」做獨立的二元分類（有麵 vs 無麵）。偵測到麵條則直接回傳 `is_lu_rou_fan=False`，不進入後續比較。

**為何不用負向提示詞列舉菜名**：牛肉麵、排骨麵、蚵仔麵線……列不完，且 CLIP 不擅長否定語意。「有麵條」這個視覺特徵是魯肉飯的硬性排除條件，適合做 gate。

CLIP 實測：所有魯肉飯照片的 noodle_prob 最高為 0.409，牛肉麵為 0.642，0.6 作為 threshold 有充裕的分隔帶。

### Tie Margin 容差設計

平手判定改為：`max_norm_vote - store_norm_vote <= _TIE_MARGIN`，預設 0.15。

**為何選 0.15**：實測中明志 vs 龍記的正規化票數差距為 0.10（0.500 vs 0.400），0.15 可覆蓋此案例並留有緩衝，同時不至於將差距明顯的情況也納入平手。

### 非魯肉飯 Hardcode 回應

`is_lu_rou_fan=False` 時直接回傳固定文字，不呼叫 Claude API。

**為何 hardcode 而非繼續呼叫 API**：原有設計中 `is_lu_rou_fan=False` 時傳給大叔的 context 描述不準確（「只看到白飯」實際上是牛肉麵），導致大叔亂掰。固定文字「所以我說，那個滷肉呢？大叔千里迢迢來鑑定，你給我看這個？」語義正確且節省 API 呼叫。

### 平手配料過濾：Known Toppings 交集

平手時，對所有候選店家的 `known_toppings` 取交集，作為 CLIP 偵測的過濾條件（邏輯與非平手時一致）。

**為何用交集而非聯集**：大叔不確定是哪家，只能回報「兩家都確認有的配料」才算有根據。聯集會把任一家的配料都報出來，反而不誠實。如果候選店家都沒有記錄 `known_toppings`，則不過濾，維持原有信任 CLIP 的邏輯。

## Risks / Trade-offs

- [Risk] noodle_prob threshold 0.6 若照片中麵條不明顯（如只露出一小角）可能誤放通 → 目前資料集中尚未出現此案例，暫不處理
- [Risk] tie_margin 0.15 對大型索引（店家數多）可能過於寬鬆，導致正常情況也觸發平手 → 若未來店家數超過 30 家需重新評估
- [Trade-off] hardcode 非魯肉飯回應犧牲了語境彈性（大叔無法針對「拍蛋糕」vs「拍牛排」說不同的話），換取準確性和零 API 成本
