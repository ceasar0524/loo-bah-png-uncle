## Context

本模組是整個系統的「準備階段」，需要在第一次使用辨識器之前先跑一次，把 30 家店家的照片全部轉為 CLIP 向量並存檔。之後每次辨識查詢都從這個索引讀取，不需要重新處理照片。

## Goals / Non-Goals

**Goals:**

- 一次性建立向量索引，後續查詢不重複處理照片
- 儲存所有照片向量與店家標籤，供 KNN 比對使用
- CPU 可跑，不需要 GPU

**Non-Goals:**

- 不做增量更新（新增店家時重建整個索引即可）
- 不做即時索引

## Decisions

### 使用 CLIP（openai/clip-vit-base-patch32）產生 embedding

CLIP 的視覺 encoder 對食物照片有良好的語意理解，且支援 zero-shot，與視覺辨識模組共用同一個模型，不需要額外載入。

### 索引以 .npz 格式持久化

使用 numpy 的 `.npz` 格式儲存所有向量、店家標籤、每家店 photo_count。格式輕量、不需要額外依賴、載入速度快。

替代方案：FAISS 向量資料庫。否決原因：數百張照片規模不需要專用向量資料庫，numpy 線性搜尋就夠快。

### 儲存所有照片向量供 KNN 比對，不取 centroid

保留每張照片的個別向量與店家標籤，供 store-matching 模組做 KNN 比對（找最像的 K 張，多數決定店家）。不取 centroid 的原因是：少數拍攝品質差的照片會拉偏平均向量，KNN 多數決更穩健。未來新增店家只需重建整個索引（偶爾新增幾家，重建一次幾分鐘，可接受）。

### 不支援的格式和損壞照片跳過不中斷

建索引時若遇到非 JPEG/PNG 或無法讀取的照片，記錄警告並繼續，不中斷整個建索引流程。

## Risks / Trade-offs

- **KNN 準確度受照片品質影響** → 照片品質差的仍會參與比對，但多數決機制可降低影響
- **建索引時間** → 400 張照片 CPU 大約幾分鐘，一次性可接受
