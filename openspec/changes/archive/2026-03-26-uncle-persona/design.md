## Context

本模組是魯肉飯辨識器 pipeline 的輸出層，負責將結構化的視覺分析結果與店家比對資訊轉化為有個性的自然語言。此模組是系統中唯一呼叫外部 LLM API 的地方，也是決定使用者體驗好壞的關鍵。

大叔人設背景：一位從年輕就愛吃魯肉飯的吃貨，年輕時夢想吃遍全台好吃的魯肉飯。隨著年紀越大、肚子越大，抱著「現在不吃以後就沒機會」的心態，已完成 30 天吃完 30 家魯肉飯的挑戰，未來目標是挑戰兩百、三百家，吃遍全台灣。

## Goals / Non-Goals

**Goals:**

- 產出具備台式幽默、文化哏、諧音梗的繁體中文回應
- 支援透過外部 JSON 範例檔調整 persona 風格，不需修改程式碼
- 處理有店家比對與無店家比對兩種情境的差異化回應

**Non-Goals:**

- 不做多輪對話或記憶使用者偏好
- 不支援其他語言輸出
- 不做回應快取或離線模式

## Decisions

### 使用 Claude API 搭配 few-shot prompting 實作大叔 persona

系統 prompt 定義大叔的身份背景與語氣規則，few-shot 範例示範具體的幽默風格（如「香菜 → 士可殺不可魯」）。LLM 從範例中學習頻率，不需要 fine-tune。

替代方案：規則式模板填充。否決原因：諧音梗和文化哏無法靠規則窮舉，LLM 生成更自然且可擴充。

### Few-shot 範例從外部 JSON 檔載入，內建 fallback

範例以 JSON 格式存於 `examples/uncle_examples.json`，格式為 `[{"input": "...", "output": "..."}]`。模組初始化時讀取；若檔案不存在則使用硬編碼的 3 條預設範例繼續運作。

替代方案：將範例硬編碼在 prompt 字串中。否決原因：調整 persona 時需改程式碼，維護成本高。

### 視覺結果與比對結果以結構化文字注入 prompt

將輸入 dict 格式化為清晰的條列文字（例如「配料：有香菜、有滷蛋；肉切法：手切塊；最相似店家：阿財（相似度 0.82）」）後插入 user message，讓 LLM 有明確事實依據產出回應。

### API 失敗回傳大叔風格 fallback，不拋出例外

API 失敗時回傳有個性的繁體中文訊息（如「大叔出去買魯肉飯，等一下」），不重試、不拋出例外。本機工具使用量低，重試機制複雜度不值得。

### 回應長度以 prompt 指定 50~80 字，並設 max_tokens 保險

prompt 明確要求大叔回應 50~80 字，同時設定 max_tokens 作為硬性上限，防止 LLM 不遵守 prompt 指示時輸出過長。

### 內容安全防護使用 prompt 層規則，保留升級至輸出後檢查的空間

現階段在 system prompt 定義五類禁止內容規則（仇恨、侮辱、性愛、暴力、不法行為），靠 LLM 自律。若未來公開給更多使用者，可升級為輸出後另呼叫一次 API 做分類檢查。

### 照片數量反映大叔熟悉度語氣，而非輸出數字

店家照片數量不直接顯示，改以「這家我很熟」vs「這家我不常去」傳遞信心差異，維持大叔人設的一致性。

## Risks / Trade-offs

- **API 費用** → 每次查詢呼叫一次 Claude API，本機工具使用量低，可接受
- **回應品質依賴 prompt 設計** → 建立範例庫持續迭代，初版先跑通流程再調整風格
- **網路連線依賴** → 本版本不做離線 fallback，使用者需自行確保 API key 與網路可用
