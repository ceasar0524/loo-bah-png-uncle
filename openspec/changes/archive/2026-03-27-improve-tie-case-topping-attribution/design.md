## Context

平手情境下，大叔取所有候選店 known_toppings 的**聯集**作為可說配料（已在前一個 change 從交集改為聯集）。但輸出到 prompt 的格式仍是「有 香菜（阿興 的特色）」，導致模型將條件式配料當成事實描述，說出「那個香菜擺在那，士可殺不可魯啦！」而非「如果是阿興那碗，可能有香菜」。

## Goals / Non-Goals

**Goals:**
- `_format_input()` 在 tie 時，將配料分成「兩家都有」與「若是 XX 那碗：可能有 YY」兩類格式輸出
- system prompt 加入說明，要求模型遇到「若是 XX 那碗」格式時使用條件語氣
- 非 tie 情境不受影響

**Non-Goals:**
- 不改變 tie 使用聯集的策略
- 不修改 store matching 邏輯

## Decisions

### 從資料格式而非 prompt 指令強制條件語氣

先前嘗試在 system prompt 加「嚴禁自行猜測香菜」的文字規則，但模型不穩定遵循。根本原因是輸入格式本身就寫「有 香菜」，模型自然當成事實。

改為在輸入資料結構上直接區分：
- 共有配料：`兩家都有的配料：肉鬆`
- 獨有配料：`若是阿興那碗：可能有 香菜（非確定，視店家而定）`

這樣模型從資料結構本身就理解哪些是條件式，不需要靠 prompt 指令強制。

### system prompt 補充配料標注說明

在「重要」規則後加：「配料標注說明：若出現『若是 XX 那碗：可能有 YY』格式，回應中提及該配料時必須使用條件語氣（如果是 XX 的話），不可當成確定事實。」作為雙重保障。

## Risks / Trade-offs

- 輸入格式變複雜，但只影響 tie 情境（佔少數情況）
- 若兩家都有某配料（如兩家都加肉鬆），仍以「兩家都有」直述，無副作用
